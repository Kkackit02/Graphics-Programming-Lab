# Vulkan 파이프라인 완벽 가이드: 래스터화와 레이 트레이싱 (최종 심층판)

이 문서는 Vulkan의 두 가지 주요 파이프라인, 즉 **그래픽스(래스터화) 파이프라인**과 **레이 트레이싱 파이프라인**의 전체 흐름과 각 단계의 역할을 상세히 설명합니다. 각 단계와 관련된 핵심 Vulkan 구조체 및 고급 개념에 대한 주석을 추가하여 실용성을 극대화했습니다.

---

### 파이프라인이란 무엇인가? (공장 조립 라인)

그래픽스 파이프라인은 3D 모델 데이터(원재료)를 입력받아 최종적으로 화면에 표시될 2D 이미지(완제품)를 만들어내는 **'공장 조립 라인'**에 비유할 수 있습니다. 각 단계는 특정 작업을 수행하며, 이 단계들이 순서대로 연결되어 하나의 파이프라인을 이룹니다.

Vulkan의 가장 큰 특징은 이 '조립 라인'의 설계를 개발자가 매우 상세하게 직접 제어해야 한다는 것입니다. 이렇게 한 번 최적화된 라인을 구축하면, 렌더링 중에는 설정을 바꾸지 않고 대량의 데이터를 매우 빠르게 처리할 수 있습니다.

---

### 1. 그래픽스 파이프라인 (Graphics/Rasterization Pipeline)

가장 전통적이고 널리 사용되는 파이프라인으로, 3D 모델을 2D 픽셀의 집합으로 변환(래스터화)하는 방식입니다.

#### **파이프라인의 각 단계 (공정)**

1.  **입력 조립기 (Input Assembler)**
    - **역할**: 버텍스 버퍼와 인덱스 버퍼에서 정점 데이터를 읽어와 파이프라인에 공급합니다.
    > **관련 구조체**: `VkPipelineInputAssemblyStateCreateInfo`
    > - `topology`: `VK_PRIMITIVE_TOPOLOGY_TRIANGLE_LIST` (기본 삼각형), `_LINE_LIST` (선) 등을 설정하여 기본 도형의 종류를 결정합니다.
    > - `primitiveRestartEnable`: `VK_TRUE`로 설정하면, 인덱스 버퍼에서 특정 값(예: `0xFFFF`)을 만나면 도형 그리기를 잠시 멈추고 다음 도형을 새로 시작할 수 있습니다.

2.  **정점 셰이더 (Vertex Shader)**
    - **역할**: 각 정점의 위치를 3D 월드 공간에서 2D 화면 공간으로 변환하는 등, 정점 단위의 연산을 수행합니다.
    > **관련 구조체**: `VkPipelineShaderStageCreateInfo`
    > - `stage`: `VK_SHADER_STAGE_VERTEX_BIT`로 설정하여 이 셰이더가 정점 셰이더임을 명시합니다.
    > - `module`: `vkCreateShaderModule`로 생성한 셰이더 모듈 핸들. 셰이더 코드는 SPIR-V라는 중간 언어로 미리 컴파일되어 있어야 합니다.
    > - `pName`: 셰이더 코드 내의 진입점 함수 이름. 보통 `"main"`입니다.

3.  **테셀레이션 & 지오메트리 셰이더 (선택 사항)**
    - **역할**: 디테일을 동적으로 추가하거나(테셀레이션), 기본 도형을 변형/생성(지오메트리)합니다.

4.  **래스터화 (Rasterization)**
    - **역할**: 벡터 형태의 도형을 픽셀 집합(프래그먼트)으로 변환합니다.
    > **관련 구조체**: `VkPipelineRasterizationStateCreateInfo`
    > - `polygonMode`: `VK_POLYGON_MODE_FILL` (면 채우기), `_LINE` (와이어프레임) 등을 설정하여 렌더링 스타일을 바꿀 수 있습니다. 디버깅에 매우 유용합니다.
    > - `cullMode`: `VK_CULL_MODE_BACK_BIT`로 설정하면 삼각형의 뒷면을 그리지 않아(컬링) 성능을 향상시킵니다.

5.  **프래그먼트 셰이더 (Fragment Shader)**
    - **역할**: 각 프래그먼트의 최종 색상을 결정합니다. 텍스처 샘플링, 조명 계산 등이 이루어집니다.

6.  **출력 병합 (Color Blending)**
    - **역할**: 프래그먼트 셰이더의 결과 색상을 프레임버퍼의 기존 색상과 혼합합니다. (알파 블렌딩 등)
    > **관련 구조체**: `VkPipelineColorBlendStateCreateInfo` 및 `VkPipelineColorBlendAttachmentState`
    > - `blendEnable`: `VK_TRUE`로 설정하면 블렌딩을 활성화합니다.
    > - `srcColorBlendFactor`, `dstColorBlendFactor`: 소스(새로 그릴 색)와 대상(이미 있는 색)의 혼합 비율을 결정합니다. (예: `VK_BLEND_FACTOR_SRC_ALPHA`, `VK_BLEND_FACTOR_ONE_MINUS_SRC_ALPHA`는 일반적인 알파 블렌딩 공식입니다.)

### 2. 고급 그래픽스 파이프라인 개념

#### **파이프라인 레이아웃 (`VkPipelineLayout`)**

- **역할**: 파이프라인과 외부 리소스(디스크립터 셋, 푸시 상수) 간의 인터페이스를 정의합니다.
- **비유**: **'공장 조립 라인의 연결 규격서'**. "우리 라인은 A 규격의 부품 공급처(Descriptor Set 0)와 B 규격의 전력선(Descriptor Set 1)을 사용하며, 작업자에게 직접 지시할 수 있는 비상 버튼(Push Constant)이 있다"고 명시합니다. 이 규격서 없이는 파이프라인이 어떤 외부 데이터를 참조해야 할지 알 수 없습니다.
- **생성**: `VkDescriptorSetLayout`과 `VkPushConstantRange` 배열로부터 `vkCreatePipelineLayout` 함수를 통해 생성됩니다.

#### **푸시 상수 (Push Constants)**

- **역할**: 매우 적은 양의 유니폼 데이터를 디스크립터 셋을 거치지 않고, 커맨드 버퍼를 통해 직접, 그리고 매우 빠르게 셰이더에 전달하는 방법입니다.
- **비유**: **'작업자에게 외치는 간단한 지시'**. 중앙 게시판(유니폼 버퍼)에 공지를 올리는 대신, 감독관이 조립 라인의 작업자에게 "이번 부품은 빨간색으로 칠해!"라고 직접 소리치는 것과 같습니다. 매우 빠르지만, 전달할 수 있는 정보의 양이 매우 제한적(보통 128바이트 내외)입니다.
- **사용**: `vkCmdPushConstants` 함수를 통해 커맨드 버퍼 기록 시 값을 지정합니다.

#### **동적 상태 (Dynamic State)**

- **역할**: 파이프라인은 대부분의 상태가 '고정'되어 있지만, 일부 상태는 '동적'으로 변경할 수 있도록 예외를 허용합니다.
- **왜 필요한가?**: 뷰포트 크기나 가위(Scissor) 영역은 창 크기가 변할 때마다 바뀌어야 합니다. 만약 이 상태들이 고정이라면, 창 크기가 변할 때마다 비싼 파이프라인 객체 전체를 다시 만들어야 합니다. 이는 매우 비효율적입니다.
- **설정**: `VkPipelineDynamicStateCreateInfo` 구조체에 동적으로 변경할 상태(예: `VK_DYNAMIC_STATE_VIEWPORT`, `VK_DYNAMIC_STATE_SCISSOR`) 목록을 지정하여 파이프라인을 생성합니다.
- **사용**: 이렇게 생성된 파이프라인은 커맨드 버퍼 기록 시 `vkCmdSetViewport`, `vkCmdSetScissor` 같은 함수를 통해 해당 상태를 실시간으로 변경할 수 있습니다.

#### **파이프라인 캐시 (`VkPipelineCache`)**

- **역할**: 한 번 생성된 파이프라인의 컴파일 결과를 저장하고 재사용하여, 다음 실행 시 파이프라인 생성 속도를 대폭 향상시킵니다.
- **왜 필요한가?**: `vkCreateGraphicsPipelines`는 매우 복잡하고 무거운 연산입니다. 드라이버는 셰이더 코드를 타겟 GPU에 맞게 최적화하고 모든 상태를 결합하여 최종 기계어를 생성해야 합니다.
- **작동 방식**:
    1.  최초 실행 시, 파이프라인을 생성하고 그 결과를 `VkPipelineCache` 객체에 저장합니다.
    2.  애플리케이션 종료 전에 `vkGetPipelineCacheData`로 캐시 데이터를 바이너리 형태로 추출하여 파일에 저장합니다.
    3.  다음 실행 시, 파일에서 캐시 데이터를 읽어 `VkPipelineCache`를 생성하고, 이 캐시를 `vkCreateGraphicsPipelines`에 전달합니다. 드라이버는 이 캐시를 보고 파이프라인을 새로 컴파일하는 대신, 기존 결과를 재사용하여 매우 빠르게 객체를 생성합니다.

### 3. 레이 트레이싱 파이프라인 (Ray Tracing Pipeline)

- **개념**: 빛의 경로를 추적하여 이미지를 생성하는 방식으로, 래스터화와는 근본적으로 다릅니다. '탐정 사무소' 비유처럼, 각 픽셀에서 쏜 광선(탐정)이 어떤 물체(단서)를 만나는지에 따라 색상을 결정합니다.
- **주요 셰이더**:
    - **Ray Generation (`.rgen`)**: '사건 담당관'. 광선 추적을 시작.
    - **Closest Hit (`.rchit`)**: '결정적 증거 분석가'. 광선이 물체와 부딪혔을 때 색상 계산.
    - **Miss (`.rmiss`)**: '미제 사건 보고관'. 광선이 아무것과도 부딪히지 않았을 때 배경색 계산.
- **생성 방식 차이**:
    - 그래픽스 파이프라인은 `vkCreateGraphicsPipelines` 함수로 생성되며, 하나의 거대한 `VkGraphicsPipelineCreateInfo` 구조체를 사용합니다. 모든 상태가 한 번에 '베이킹'됩니다.
    - 레이 트레이싱 파이프라인은 `vkCreateRayTracingPipelinesKHR` 함수로 생성되며, 여러 개의 셰이더 그룹(`VkRayTracingShaderGroupCreateInfoKHR`)의 배열을 입력으로 받습니다. 이는 래스터화처럼 고정된 스테이지 흐름이 아니라, 다양한 셰이더들이 이벤트 기반으로 호출되는 레이 트레이싱의 동적인 특성을 반영합니다.

---

### 4. 두 파이프라인의 비교 요약

| 항목 | 그래픽스 (래스터화) 파이프라인 | 레이 트레이싱 파이프라인 |
| :--- | :--- | :--- |
| **기본 원리** | 3D 모델 -> 2D 픽셀로 **투영 및 색칠** | 2D 픽셀 -> 3D 씬으로 **광선 추적** |
| **강점** | **속도**. 대부분의 실시간 렌더링에 사용. | **사실감**. 반사, 굴절, 부드러운 그림자 표현에 탁월. |
| **핵심 단계** | 정점 셰이더 -> 래스터화 -> 프래그먼트 셰이더 | 광선 생성 -> 교차 테스트 -> Hit/Miss 셰이더 |
| **주요 데이터** | 정점/인덱스 버퍼 | 가속 구조 (AS), 셰이더 바인딩 테이블 (SBT) |

현대의 고성능 애플리케이션은 두 파이프라인을 혼합하여 사용하는 **하이브리드 렌더링(Hybrid Rendering)**을 통해 속도와 사실감을 모두 잡으려는 경향이 있습니다.

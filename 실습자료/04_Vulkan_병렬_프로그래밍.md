# Vulkan 학습 자료집: 04. 병렬 프로그래밍 (심층 보강판)

이 문서는 `수정_04Vulkan (2).pdf` 파일의 내용을 바탕으로, Vulkan의 가장 강력한 기능 중 하나인 **병렬 프로그래밍**에 대해 '총괄 셰프와 부주방장' 비유를 들어 더 쉽고 깊이 있게 설명합니다. 멀티코어 CPU를 최대한 활용하여 렌더링 성능을 극대화하는 원리와 구현 방법을 학습합니다.

---

### 1. 학습 목표

- **CPU 병목 현상과 Vulkan의 해법 이해**: 고성능 GPU가 왜 CPU 때문에 제 성능을 못 내는지, Vulkan이 이 문제를 어떻게 해결하는지 명확히 이해합니다.
- **스레드 안전성과 커맨드 풀**: '개인용 작업대' 비유를 통해 왜 스레드마다 별도의 커맨드 풀이 필요한지 각인합니다.
- **Primary/Secondary 커맨드 버퍼의 역할극**: '총괄 셰프'와 '부주방장'의 역할 분담을 통해 두 커맨드 버퍼의 관계를 명확히 이해합니다.
- **멀티스레드 렌더링의 전체 시나리오 파악**: 여러 스레드가 협력하여 하나의 프레임을 완성하는 구체적인 과정을 학습합니다.

---

### 2. 왜 병렬 프로그래밍이 필요한가? (셰프와 인쇄 공장)

- **비유**: GPU는 1초에 책 1,000권을 인쇄할 수 있는 **'초고속 인쇄 공장'**입니다. 하지만 '사무실'에 있는 **CPU라는 '작가'**가 1초에 책 10권 분량의 원고(렌더링 명령)만 겨우 작성해서 넘겨준다면, 공장은 대부분의 시간 동안 놀게 됩니다. 이것이 바로 **CPU 병목(CPU-bound)** 현상입니다.

- **Vulkan의 해법**: 작가 한 명이 아니라, **여러 명의 작가(CPU 코어)가 동시에 원고의 각 챕터(씬의 일부)를 작성**해서 인쇄 공장을 쉬지 않고 돌리는 것입니다. Vulkan은 이러한 병렬 원고 작성을 위해 설계된 API입니다.

---

### 3. 커맨드 풀 (`VkCommandPool`)과 스레드 안전성: 개인용 작업대

- **핵심 문제**: `VkCommandPool`은 **'하나뿐인 공유 작업대'**와 같습니다. 여러 직원(스레드)이 동시에 이 작업대를 사용하여 도구(커맨드 버퍼)를 꺼내거나 정리하려고 하면 서로 부딪히고 엉망이 됩니다.
- **잘못된 접근**: 이 작업대에 '자물쇠(Mutex)'를 걸어 한 번에 한 명만 쓰도록 할 수 있습니다. 하지만 이러면 직원들은 결국 줄을 서서 기다려야 하므로, 여러 명이 있는 의미가 퇴색됩니다.
- **Vulkan의 해결책**: 각 직원에게 **'개인용 작업대(`VkCommandPool`)'**를 하나씩 나눠줍니다. 이제 각 직원은 다른 사람 눈치 볼 필요 없이 자신의 작업대에서 자유롭게 도구를 꺼내고(커맨드 버퍼 할당) 작업을 준비(명령 기록)할 수 있습니다. 이것이 바로 **"스레드당 하나의 커맨드 풀(One Command Pool per Thread)"** 패턴입니다.

---

### 4. Primary vs. Secondary 커맨드 버퍼: 총괄 셰프와 부주방장

Vulkan의 병렬 작업을 이해하기 위해 '레스토랑 주방' 비유를 사용해 봅시다.

1.  **Primary 커맨드 버퍼 (`VK_COMMAND_BUFFER_LEVEL_PRIMARY`) - 총괄 셰프**
    - **역할**: 코스 요리의 전체 흐름을 지휘하는 **'총괄 셰프(Master Chef)'**입니다.
    - **권한**:
        - 주방의 문을 열고 닫을 수 있습니다 (`vkCmdBeginRenderPass`, `vkCmdEndRenderPass`).
        - 최종 완성된 요리를 손님(GPU 큐)에게 직접 내보낼 수 있습니다 (`vkQueueSubmit`).
        - 여러 부주방장들의 작업을 취합하여 최종 요리를 완성하라고 지시할 수 있습니다 (`vkCmdExecuteCommands`).

2.  **Secondary 커맨드 버퍼 (`VK_COMMAND_BUFFER_LEVEL_SECONDARY`) - 부주방장**
    - **역할**: 각자 전문 분야(에피타이저, 메인, 디저트)를 담당하는 **'부주방장(Sous-Chef)'**들입니다.
    - **권한**:
        - 자신에게 할당된 요리(씬의 일부 렌더링)를 독립적으로 준비(명령 기록)할 수 있습니다.
        - 하지만, 주방 문을 직접 열거나 닫을 수 없으며, 손님에게 직접 요리를 내보낼 수도 없습니다. 모든 작업은 반드시 총괄 셰프의 지휘를 통해 이루어져야 합니다.

> **심층 주석: 상태 상속(State Inheritance)의 함정**
> `vkCmdExecuteCommands`를 호출할 때, Primary Command Buffer의 상태(예: 바인딩된 파이프라인, 디스크립터 셋)는 Secondary Command Buffer로 **상속되지 않습니다.** 각 Secondary Command Buffer는 자신이 사용할 파이프라인과 디스크립터 셋을 `vkCmdBindPipeline`, `vkCmdBindDescriptorSets`를 통해 직접 바인딩해야 합니다. 이는 각 Secondary Command Buffer가 완전히 독립적인 렌더링 작업 단위임을 보장하여, 병렬로 기록할 때 생길 수 있는 복잡한 의존성 문제를 원천적으로 차단합니다.

#### **`VkCommandBufferInheritanceInfo`: 총괄 셰프의 작업 지침서**

- **역할**: 총괄 셰프가 부주방장들에게 "오늘 저녁 파티는 5번 테이블에서 진행되고, 메인 요리는 스테이크야. 여기에 맞춰서 각자 요리 준비해!" 라고 미리 알려주는 **'작업 지침서'**와 같습니다.
- **설명**: 이 구조체는 보조 커맨드 버퍼가 어떤 렌더 패스와 프레임버퍼의 컨텍스트 내에서 실행될 것인지에 대한 정보를 담습니다. 부주방장(Secondary Buffer)은 이 지침서를 보고 "아, 5번 테이블용 샐러드를 만들면 되겠구나" 하고 자신의 작업을 전체 흐름에 맞게 준비할 수 있습니다. 이 정보가 없으면 드라이버는 해당 명령이 유효한지 검증할 수 없으며, Secondary Command Buffer는 유효하게 기록될 수 없습니다.

### 5. 멀티스레드 렌더링 시나리오: 저녁 코스 요리 준비하기

1.  **총괄 셰프 (메인 스레드) - 작업 준비**
    - "저녁 장사 시작!" -> `vkBeginCommandBuffer`로 자신의 작업(Primary Buffer)을 시작합니다.
    - "5번 테이블 코스 요리 준비!" -> `vkCmdBeginRenderPass`로 주방 문을 엽니다. 이때, "요리는 부주방장들이 만들 거야(`VK_SUBPASS_CONTENTS_SECONDARY_COMMAND_BUFFERS`)"라고 선언합니다.
    - "자, 에피타이저 팀, 메인 팀, 디저트 팀! 각자 요리 시작해!" -> 스레드 풀에 작업을 분배합니다.

2.  **부주방장들 (워커 스레드) - 병렬 요리**
    - 각자 자신의 개인 작업대(`VkCommandPool`)로 갑니다.
    - 총괄 셰프의 지침서(`VkCommandBufferInheritanceInfo`)를 확인합니다.
    - 각자 맡은 요리(샐러드, 스테이크, 케이크)를 레시피에 따라 만듭니다(렌더링 명령 기록).
    - 요리 준비가 끝나면 총괄 셰프에게 보고합니다(스레드 작업 종료).

3.  **총괄 셰프 (메인 스레드) - 최종 완성 및 서빙**
    - "모든 팀 준비 완료됐나?" -> 모든 스레드가 작업을 마칠 때까지 기다립니다.
    - 준비된 샐러드, 스테이크, 케이크(Secondary Buffers)를 순서대로 접시에 올립니다. -> `vkCmdExecuteCommands`를 호출하여 준비된 모든 보조 커맨드 버퍼를 실행하라고 지시합니다.
    - "5번 테이블 요리 끝!" -> `vkCmdEndRenderPass`로 주방 문을 닫습니다.
    - "서빙!" -> 완성된 작업 지시서(Primary Buffer)를 웨이터(GPU 큐)에게 전달합니다 (`vkQueueSubmit`).

---

### 6. 시험 대비 핵심 요약

- **성능 문제**: 복잡한 씬에서는 CPU가 GPU에게 줄 명령을 만드는 속도가 느려서 GPU가 놀게 된다 (CPU 병목).
- **Vulkan의 해법**: 여러 CPU 코어(스레드)가 렌더링 명령을 동시에 만들게 하자!
- **핵심 규칙**: **스레드 안전성을 위해, 각 스레드는 자신만의 `VkCommandPool`을 가져야 한다.**
- **역할 분담**:
    - **Primary Buffer (총괄 셰프)**: 전체 흐름 제어, 큐에 직접 제출.
    - **Secondary Buffer (부주방장)**: 실제 작업 내용 기록, 병렬 생성 담당, Primary에 의해 실행됨.
- **핵심 API 및 파라미터**:
    - `vkAllocateCommandBuffers`의 `level` 파라미터: `PRIMARY`와 `SECONDARY`를 구분.
    - `VkCommandBufferInheritanceInfo`: **(Secondary 전용)** 부모(Primary)로부터 어떤 상태(렌더 패스 등)를 상속받을지 명시.
    - `vkCmdExecuteCommands`: **(Primary 전용)** 여러 Secondary 커맨드 버퍼들을 실행시키는 명령.
    - `vkCmdBeginRenderPass`의 `contents` 파라미터:
        - `VK_SUBPASS_CONTENTS_INLINE`: **(기본값)** 모든 그리기 명령이 Primary 버퍼에 직접 기록됨.
        - `VK_SUBPASS_CONTENTS_SECONDARY_COMMAND_BUFFERS`: 그리기 명령이 Secondary 버퍼들로 제공될 것임을 드라이버에 알려, 병렬 처리에 최적화되도록 함.

이 '주방' 비유를 통해, Vulkan의 병렬 처리 구조가 단순히 코드를 나누는 것이 아니라, 각자의 역할과 책임을 명확히 하여 효율성을 극대화하는 체계적인 시스템이라는 것을 이해할 수 있습니다.

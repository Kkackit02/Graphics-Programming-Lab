# Vulkan 학습 자료집: 01. 소개 및 기본 설정 (상세 설명판)

이 문서는 `수정_01Vulkan (2).pdf` 파일의 내용을 바탕으로, 복잡한 Vulkan의 개념들을 더 쉽게 이해할 수 있도록 **다양한 비유와 상세한 부연 설명**을 추가한 자료입니다. 각 단계에서 "왜?"라는 질문에 답하며 개념을 깊이 있게 파악하고, 시험 대비용 참고 자료로 활용할 수 있도록 구성되었습니다.

### 1. 학습 목표

- **Vulkan의 핵심 철학 이해**: Vulkan이 왜 빠르고, 왜 복잡한지에 대한 근본적인 이유를 파악합니다.
- **Vulkan 기본 구성 요소 마스터**: `인스턴스`, `디바이스`, `스왑체인` 등 필수 개념들을 비유를 통해 명확히 이해합니다.
- **개발 환경 설정 완전 정복**: Visual Studio에서 Vulkan 프로젝트를 설정하는 각 단계의 의미를 이해하고 실습합니다.
- **첫 번째 삼각형 렌더링**: Vulkan으로 화면에 무언가를 그리는 전체 과정을 단계별로 따라가며 흐름을 익힙니다.

### 2. Vulkan이란 무엇인가? (쉽게 이해하기)

- **Vulkan**: 고성능을 목표로 하는 차세대 그래픽스 및 컴퓨팅 API입니다.
- **핵심 특징: 낮은 오버헤드 (Low Overhead)**
    - **비유**: 숙련된 요리사에게 "스테이크 하나"라고만 말하는 것(OpenGL/DirectX)과, "팬 예열은 3분, 고기는 센 불에 각 면 1분씩, 레스팅은 5분"이라고 모든 과정을 직접 지시하는 것(Vulkan)의 차이입니다.
    - **설명**: Vulkan은 드라이버가 알아서 해주던 많은 작업(메모리 관리, 오류 검사 등)을 개발자에게 위임합니다. 이로 인해 드라이버의 부담이 줄어(오버헤드 감소), CPU 자원을 더 효율적으로 사용하고 멀티코어 CPU를 최대한 활용하여 GPU에 렌더링 명령을 더 빠르게 전달할 수 있습니다.
- **장점**: 성능이 매우 뛰어나고, 여러 CPU 코어를 효율적으로 활용할 수 있습니다.
- **단점**: 코드가 매우 장황하고, 모든 것을 직접 제어해야 하므로 배우기 어렵습니다. 작은 실수 하나가 치명적인 오류로 이어질 수 있습니다.

### 3. Vulkan 개발 환경 설정 (Visual Studio)

Vulkan 프로그램을 만들려면, 우리 컴퓨터에 Vulkan이 있다는 것을 컴파일러와 링커에게 알려주어야 합니다.

#### **1단계: 필요 라이브러리 다운로드**

1.  **Vulkan SDK**: Vulkan 개발에 필요한 모든 도구, 라이브러리, 헤더 파일이 포함된 공식 개발 키트입니다. **vkcube.exe** 실행으로 설치를 확인합니다.
2.  **GLFW**: 화면에 창(Window)을 만들고, 키보드/마우스 입력을 처리해주는 라이브러리입니다. Vulkan은 창 생성 기능이 없어서 GLFW 같은 보조 라이브러리가 필수입니다.
3.  **GLM**: 벡터, 행렬 등 그래픽스 수학 연산을 쉽게 해주는 헤더 전용 라이브러리입니다.

#### **2단계: Visual Studio 프로젝트 설정**

**왜 이 설정이 필요한가?**
컴퓨터는 우리가 작성한 C++ 코드를 스스로 이해하지 못합니다. 컴파일러가 코드를 기계어로 번역하고, 링커가 여러 코드 조각과 라이브러리를 합쳐 최종 실행 파일을 만듭니다. 이 과정에서 필요한 파일들이 어디 있는지 알려주는 것이 바로 프로젝트 설정입니다.

1.  **C/C++ -> 일반 -> 추가 포함 디렉터리**:
    - **무엇을**: Vulkan, GLFW, GLM의 `include` 또는 `header` 폴더 경로를 추가합니다.
    - **왜?**: 코드에서 `#include <vulkan/vulkan.h>` 같은 구문을 만났을 때, 컴파일러가 해당 헤더 파일을 찾을 수 있도록 경로를 알려줍니다. 헤더 파일에는 함수의 선언, 구조체 정의 등이 들어있습니다.

2.  **링커 -> 일반 -> 추가 라이브러리 디렉터리**:
    - **무엇을**: Vulkan, GLFW의 `lib` 폴더 경로를 추가합니다.
    - **왜?**: 헤더 파일에 선언만 있던 함수들의 실제 구현 코드가 담긴 라이브러리 파일(`.lib`)을 링커가 찾을 수 있도록 경로를 알려줍니다.

3.  **링커 -> 입력 -> 추가 종속성**:
    - **무엇을**: `vulkan-1.lib`, `glfw3.lib` 등 실제 사용할 라이브러리 파일의 이름을 직접 적어줍니다.
    - **왜?**: 위에서 알려준 경로에 수많은 `.lib` 파일이 있을 수 있습니다. 그중에서 우리 프로그램이 **정확히 어떤 라이브러리 파일과 연결(link)되어야 하는지** 링커에게 명시해주는 것입니다.

4.  **C/C++ -> 언어 -> C++ 언어 표준**:
    - **무엇을**: `C++17` 이상으로 설정합니다.
    - **왜?**: Vulkan 예제 코드들이 C++17의 최신 문법을 사용하기 때문에, 컴파일러가 이 문법을 이해하고 올바르게 번역하도록 설정합니다.

### 4. Vulkan 렌더링 파이프라인: 첫 삼각형 그리기

Vulkan으로 무언가를 그리려면 수많은 객체를 생성하고 설정하는 과정을 거쳐야 합니다. 각 단계는 매우 명확한 역할을 가집니다.

#### **렌더링을 위한 8단계 핵심 흐름**

**비유**: "삼각형 그림 그리기"라는 목표를 위해, 필요한 도구(물감, 붓, 캔버스)를 준비하고, 그림 그리는 순서를 정한 뒤, 실제로 그림을 그리고 전시하는 과정과 같습니다.

1.  **`VkInstance` 생성**: **(Vulkan 시스템 부팅)**
    - **역할**: Vulkan API를 사용하겠다고 시스템에 알리고, 드라이버와 애플리케이션을 연결합니다.
    - **비유**: 게임을 하기 위해 **게임 런처(스팀, 에픽게임즈 런처 등)를 실행**하는 것과 같습니다. 어떤 확장 기능(validation layer 등)을 사용할지 여기서 알려줍니다.

2.  **`VkPhysicalDevice` 선택 및 `VkLogicalDevice` 생성**: **(그래픽 카드 선택 및 기능 활성화)**
    - **`VkPhysicalDevice`**: 컴퓨터에 설치된 물리적인 그래픽 카드(하드웨어) 자체입니다.
    - **`VkLogicalDevice`**: 선택한 물리 그래픽 카드의 어떤 기능(예: 지오메트리 셰이더)을 사용할지 구체적으로 명시하여 생성하는 **소프트웨어 핸들**입니다.
    - **비유**: 컴퓨터에 그래픽 카드가 여러 개 있다면(`VkPhysicalDevice` 목록), 그중에서 게임을 실행할 **주 그래픽 카드를 선택**하고(`pickPhysicalDevice`), 해당 카드의 **그래픽 처리 기능을 활성화**(`createLogicalDevice`)하는 것과 같습니다.

3.  **`VkSurface` 및 `VkSwapchain` 생성**: **(캔버스와 플립북 준비)**
    - **`VkSurface`**: 렌더링 결과가 그려질 창(Window)의 표면. GLFW를 통해 생성합니다.
    - **`VkSwapchain`**: 화면에 부드러운 애니메이션을 보여주기 위한 이미지들의 큐(Queue). 보통 2개(더블 버퍼링) 또는 3개(트리플 버퍼링)의 이미지로 구성됩니다.
    - **비유**: 그림을 그릴 **캔버스(`VkSurface`)**를 이젤에 올려놓고, 부드러운 애니메이션을 위해 여러 장의 그림을 미리 그려놓고 빠르게 넘길 수 있는 **플립북(`VkSwapchain`)**을 준비하는 과정입니다.

4.  **`VkImageView` 생성**: **(그림 해석 방법 정의)**
    - **역할**: 스왑체인에 있는 `VkImage`들을 어떻게 바라보고 사용할지(예: 색상 정보로 사용, 특정 부분만 사용 등) 정의합니다.
    - **비유**: 사진 원본(`VkImage`)이 있을 때, "이 사진은 컬러 사진으로 해석해라" 또는 "사진의 가운데 부분만 잘라서 사용해라"라고 **사용 설명서**를 붙여주는 것과 같습니다.

5.  **`VkRenderPass` 생성**: **(그림 그리기 작업 명세서 작성)**
    - **역할**: 렌더링 작업의 전체 구조를 정의합니다. 몇 개의 컬러 버퍼를 사용할지, 뎁스 버퍼는 쓸 것인지, 렌더링 시작과 끝에 버퍼를 어떻게 처리할지(예: 이전 그림 지우기) 등을 명시합니다.
    - **비유**: 그림을 그리기 전에 **작업 계획서**를 작성하는 것과 같습니다. "1. 캔버스를 파란색으로 칠한다. 2. 그 위에 삼각형을 그린다. 3. 그림이 끝나면 서명한다." 와 같이 렌더링의 전체 단계를 정의합니다.

6.  **`VkPipeline` 생성**: **(공장 조립 라인 구축)**
    - **역할**: 렌더링의 모든 단계를 하나로 묶는 가장 중요하고 복잡한 객체입니다. 버텍스 데이터가 어떤 형식인지, 셰이더는 무엇을 쓸지, 도형을 어떻게 그릴지(점, 선, 면), 뷰포트 크기는 어떤지 등 렌더링에 관한 거의 모든 상태를 고정시킵니다.
    - **비유**: 자동차를 만드는 **공장 조립 라인**을 구축하는 것과 같습니다. 각 공정(셰이더, 입력 조립, 래스터화 등)을 설정하고 연결하여, 원재료(정점 데이터)가 들어오면 최종 제품(화면 픽셀)이 나올 때까지의 모든 과정을 정의합니다. 한 번 만들어지면 변경이 어려워 높은 성능을 보장합니다.

7.  **`VkFramebuffer`, `VkCommandPool`, `VkCommandBuffer` 생성**: **(작업 공간과 작업 지시서 준비)**
    - **`VkFramebuffer`**: 렌더 패스가 실제로 그림을 그릴 캔버스(이미지 뷰)들을 묶어놓은 것입니다.
    - **`VkCommandPool`**: 커맨드 버퍼를 할당하기 위한 메모리 풀. 스레드와 관련이 깊습니다.
    - **`VkCommandBuffer`**: GPU가 실행할 명령(`vkCmd...` 함수들)을 기록하는 **작업 지시서**입니다.
    - **비유**: 렌더 패스라는 **작업 계획서**에 따라, 프레임버퍼라는 **실제 캔버스** 위에서, 커맨드 버퍼라는 **작업 지시서**에 "삼각형을 그려라", "파이프라인을 사용해라" 등의 명령을 순서대로 기록하는 것입니다.

8.  **`mainLoop`에서 프레임 렌더링 및 제출**: **(작업 지시 및 전시)**
    - **`drawFrame` 함수**: 매 프레임 호출되며 다음 작업을 수행합니다.
        1.  **동기화**: GPU가 이전 프레임 작업을 마칠 때까지 기다립니다 (`vkWaitForFences`).
        2.  **이미지 획득**: 스왑체인(플립북)에서 그림을 그릴 다음 빈 종이(이미지)를 가져옵니다 (`vkAcquireNextImageKHR`).
        3.  **커맨드 버퍼 기록**: 위에서 준비한 커맨드 버퍼에 렌더링 명령들을 기록합니다 (`vkCmdBeginRenderPass`, `vkCmdBindPipeline`, `vkCmdDraw`, `vkCmdEndRenderPass`).
        4.  **큐 제출**: 명령 기록이 끝난 커맨드 버퍼를 GPU의 작업 창구(`VkQueue`)에 제출합니다 (`vkQueueSubmit`). 이제 GPU가 실제로 그림을 그리기 시작합니다.
        5.  **화면 표시(Present)**: GPU가 그림을 다 그리면, 그 결과를 화면에 보여달라고 스왑체인에 요청합니다 (`vkQueuePresentKHR`).

### 5. 시험 대비 핵심 요약

- **Vulkan의 철학**: "명령은 명시적으로, 제어는 개발자가." 드라이버를 믿지 말고 모든 것을 직접 관리하여 성능을 쥐어짜는 것이 핵심입니다.
- **초기화 순서**: `Instance` -> `PhysicalDevice` -> `LogicalDevice` -> `Surface` -> `Swapchain` -> `ImageViews` -> `RenderPass` -> `Pipeline` -> `Framebuffers` -> `CommandPool` -> `CommandBuffers`. 이 흐름은 거의 모든 Vulkan 애플리케이션의 기본 골격입니다.
- **객체 간의 관계**:
    - `VkInstance`는 모든 것의 시작점입니다.
    - `VkDevice`는 `VkPhysicalDevice`로부터 만들어지며, 대부분의 Vulkan 객체(버퍼, 이미지, 파이프라인 등)를 생성하는 주체입니다.
    - `VkPipeline`은 `VkRenderPass`와 `VkDescriptorSetLayout`에 의존합니다.
    - `VkCommandBuffer`는 `VkCommandPool`에서 할당되며, `vkCmd...` 명령을 기록합니다.
    - `VkQueue`는 `VkCommandBuffer`를 제출받아 GPU에서 실행시킵니다.
- **`drawFrame` 함수의 역할**: 동기화, 이미지 획득, 명령 기록, 큐 제출, 화면 표시 요청이라는 5가지 핵심 작업을 매 프레임 반복합니다.

이 문서를 통해 Vulkan의 복잡한 초기화 과정이 각기 어떤 의미를 가지며 왜 필요한지에 대한 감을 잡으셨기를 바랍니다. 시험 시 특정 객체의 역할이나 생성 과정이 헷갈릴 때 이 문서를 참고하시면 큰 도움이 될 것입니다.

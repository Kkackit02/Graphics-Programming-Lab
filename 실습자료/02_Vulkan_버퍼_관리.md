# Vulkan 학습 자료집: 02. 버퍼 관리 (심층 보강판)

이 문서는 `수정_02Vulkan (1).pdf` 파일의 내용을 바탕으로, Vulkan에서 데이터를 GPU로 전송하고 관리하는 핵심 요소인 **버퍼(Buffer)**와, 이를 셰이더에 연결하는 **디스크립터(Descriptor)**에 대해 비유를 통해 더 쉽게 설명합니다.

---

### 1. 학습 목표

- **버텍스 버퍼 (Vertex Buffer)**: 모델의 '뼈대'인 정점 데이터를 GPU에 어떻게 저장하고 사용하는지 원리를 파악합니다.
- **유니폼 버퍼 (Uniform Buffer)**: 모델 전체에 적용되는 '전역 설정'인 변환 행렬 등을 어떻게 전달하고 업데이트하는지 배웁니다.
- **디스크립터 (Descriptor)**: 버퍼와 셰이더를 연결하는 '주소록' 시스템의 작동 방식을 명확히 이해합니다.
- **Staging Buffer**: CPU와 GPU 간의 효율적인 데이터 전송 전략을 학습합니다.

---

### 2. 버텍스 버퍼 (Vertex Buffer): 모델의 뼈대를 GPU에 전달하기

버텍스 버퍼는 렌더링할 모델의 모든 정점(위치, 색상, 텍스처 좌표 등) 정보를 담고 있는 GPU 메모리 덩어리입니다.

- **비유**: GPU를 '조립 공장', 모델을 '로봇'이라고 한다면, 버텍스 버퍼는 로봇의 모든 부품(팔, 다리, 머리 등)의 **설계도와 재료가 담긴 '자재 창고'**와 같습니다. GPU는 이 창고에서 부품 정보를 가져와 로봇을 조립(렌더링)합니다.

> **심층 주석: 셰이더와 메모리의 연결고리**
> 셰이더 코드는 `layout(location = 0) in vec3 inPosition;` 처럼 추상적인 '입력 슬롯'만 알고 있습니다. `VkVertexInputAttributeDescription`은 "셰이더의 0번 슬롯은, 0번 바인딩(버퍼)에서 가져오는데, 데이터 형식은 `vec3`이고, 각 정점 데이터 덩어리의 맨 앞(offset 0)에서 시작해" 라고 구체적으로 지정해줍니다. `VkVertexInputBindingDescription`은 "그리고 그 0번 바인딩(버퍼)의 데이터 덩어리는 `sizeof(Vertex)` 바이트마다 하나씩 있어" 라고 알려주어 GPU가 다음 정점 데이터를 정확히 찾아가게 합니다. 이 두 구조체는 셰이더와 GPU 메모리 사이의 핵심적인 다리 역할을 합니다.

#### **Staging Buffer: 효율적인 자재 운송 시스템**

GPU가 가장 빠르게 접근할 수 있는 메모리(`DEVICE_LOCAL_BIT`)는 보통 CPU가 직접 접근할 수 없습니다. 그래서 '운송용 카트' 개념이 필요합니다.

- **Staging Buffer**: CPU가 접근 가능한 메모리. **비유하자면 'CPU의 작업대'와 GPU의 '자재 창고' 사이를 오가는 '운송용 카트'**입니다.
- **작동 방식**:
    1.  CPU가 자신의 작업 공간에서 물건(정점 데이터)을 만들어 **운송용 카트(Staging Buffer)**에 싣습니다. (`vkMapMemory`, `memcpy`)
    2.  GPU에게 "카트에 물건 실었으니, 너의 창고로 옮겨"라고 명령합니다. (`vkCmdCopyBuffer`)
    3.  GPU는 내부적으로 가장 빠른 통로를 이용해 카트의 물건을 자신의 **전용 창고(Vertex Buffer, `DEVICE_LOCAL_BIT`)**로 옮깁니다.
    4.  이후 렌더링 시, GPU는 CPU의 방해 없이 자신의 전용 창고에서 자재를 매우 빠르게 가져다 사용합니다.

> **심층 주석: 왜 Staging Buffer를 써야만 하는가?**
> GPU 메모리는 종류가 다양합니다. CPU가 쉽게 접근할 수 있는 메모리(`HOST_VISIBLE`)는 GPU 입장에서는 상대적으로 느립니다. 반면 GPU가 가장 빠르게 접근하는 고성능 메모리(`DEVICE_LOCAL`)는 CPU가 직접 접근하기 어렵거나 불가능합니다. 렌더링 루프 동안 GPU는 매 프레임 버텍스 버퍼를 읽어야 하므로, 이 버퍼는 가장 빠른 `DEVICE_LOCAL` 메모리에 있는 것이 이상적입니다. Staging Buffer는 CPU가 데이터를 써놓을 수 있는 `HOST_VISIBLE` 메모리에 임시로 버퍼를 만들고, GPU의 고속 전송 기능(`vkCmdCopyBuffer`)을 이용해 이 데이터를 `DEVICE_LOCAL` 메모리에 있는 최종 버텍스 버퍼로 '단 한 번' 복사하기 위한 전략입니다. 초기 설정은 복잡하지만, 렌더링 성능을 극대화하기 위한 필수적인 과정입니다.

### 3. 유니폼 버퍼 (Uniform Buffer): 모든 정점에 적용되는 공지사항

유니폼 버퍼는 셰이더 내 모든 정점(또는 프래그먼트)에 동일하게 적용되는 데이터를 담습니다.

- **비유**: 공장의 모든 조립 로봇(정점)들이 함께 보는 **'중앙 공지사항 게시판'**입니다. 게시판에 "오늘의 목표: 모든 로봇 팔을 오른쪽으로 5도 회전"이라고 쓰면, 모든 로봇들이 그에 맞춰 동작합니다. 모델의 전체적인 이동, 회전, 크기 변환, 카메라 시점, 조명 색상 등이 여기에 해당합니다.

#### **애니메이션의 원리: 게시판 내용 계속 바꾸기**

회전하는 사각형과 같은 애니메이션은 매 프레임마다 이 '공지사항 게시판(유니폼 버퍼)'의 내용을 새로 써주는 방식으로 구현됩니다.

- **`updateUniformBuffer` 함수의 역할**:
    1.  현재 시간을 기준으로 새로운 회전 각도를 계산합니다.
    2.  이 각도를 사용하여 새로운 모델 변환 행렬(`ubo.model`)을 만듭니다.
    3.  `memcpy`를 통해 이 새로운 행렬 데이터를 유니폼 버퍼 메모리에 덮어씁니다.
- **`vkMapMemory`**: 이 덮어쓰기 작업을 위해, CPU가 GPU의 유니폼 버퍼 메모리에 접근할 수 있는 '연결 통로'를 여는 함수입니다. 한 번 매핑해두면(`uniformBuffersMapped`), 렌더링 루프 중에는 해제 없이 계속 사용할 수 있습니다.

### 4. 디스크립터 (Descriptor): 셰이더를 위한 '주소록' 시스템

셰이더 코드는 `layout(binding = 0) uniform ...` 처럼 데이터가 필요하다고 선언만 할 뿐, 그 데이터가 실제 GPU 메모리의 어느 주소에 있는지는 모릅니다. 디스크립터는 이 둘을 연결해주는 정교한 '주소록' 시스템입니다.

- **비유**: 셰이더를 '신입사원', 버퍼를 '사내 문서'라고 해봅시다.
    - **셰이더**: "업무에 '카메라 위치' 문서가 필요해요!"
    - **Vulkan**: "그 문서는 메모리 주소 `0x1A2B3C`에 있어." (직접 주소를 알려주는 것은 위험하고 비효율적)
    - **디스크립터 방식**: "너의 '업무용 주소록(Descriptor Set)'의 '0번 항목(binding=0)'을 찾아봐. 거기에 '카메라 위치' 문서의 위치가 적혀있어."

#### **디스크립터 설정 워크플로우: 도서관 시스템 구축하기**

1.  **`VkDescriptorSetLayout` 생성 (도서관의 '장서 분류 규칙' 만들기)**
    - **역할**: 우리 파이프라인이 어떤 종류의 리소스(책)를, 몇 번 서가(binding)에서 사용할 것인지 **설계도를 정의**합니다.
    - **비유**: "우리 도서관의 0번 서가에는 '소설(UBO)'만, 1번 서가에는 '잡지(Texture Sampler)'만 꽂을 수 있다"고 규칙을 정하는 단계입니다. 이 규칙 없이는 책을 체계적으로 관리할 수 없습니다.

2.  **`VkDescriptorPool` 생성 (도서관 '건물' 짓기)**
    - **역할**: 실제 디스크립터(책갈피)들을 할당할 수 있는 **메모리 풀**을 만듭니다.
    - **비유**: "'소설책 10권, 잡지 20권을 보관할 수 있는 공간'을 가진 도서관 건물을 짓는 것입니다. 공간이 부족하면 책을 더 보관할 수 없습니다.

3.  **`vkAllocateDescriptorSets` 실행 ('회원증' 발급받기)**
    - **역할**: 위에서 만든 풀에서, 정의된 레이아웃(분류 규칙)에 맞는 **실제 디스크립터 셋을 할당**받습니다.
    - **비유**: 도서관에서 '장서 분류 규칙'에 따라 책을 빌릴 수 있는 '회원증(`VkDescriptorSet`)'을 발급받는 과정입니다.

4.  **`vkUpdateDescriptorSets` 실행 ('책'을 서가에 꽂기)**
    - **역할**: 할당된 디스크립터 셋의 각 바인딩이 **실제 어떤 리소스(버퍼, 이미지)를 가리킬지 연결**합니다.
    - **비유**: 발급받은 회원증을 사용하여, "0번 서가에는 '카메라 행렬'이라는 실제 책(버퍼)을 꽂아두겠다"고 지정하는, 가장 구체적인 연결 작업입니다.

5.  **`vkCmdBindDescriptorSets` 실행 (책 참고하여 작업하기)**
    - **역할**: 커맨드 버퍼에 "이제부터 이 디스크립터 셋(회원증)에 적힌 주소록을 참고해서 렌더링 작업을 수행해라"라고 명령을 기록합니다.
    - **비유**: 신입사원(셰이더)에게 업무에 필요한 문서들이 정리된 주소록(디스크립터 셋)을 건네주는 것과 같습니다.

> **심층 주석: 디스크립터와 파이프라인 레이아웃**
> `VkDescriptorSetLayout`은 파이프라인을 만들 때 사용되는 `VkPipelineLayout`을 생성하는 데에도 사용됩니다. `VkPipelineLayout`은 "우리 파이프라인(공장 라인)은 앞으로 0번 서가(binding 0)의 책(UBO)을 참고해서 작업할 것이다"라고 공장 라인 자체에 명시하는 것과 같습니다. 나중에 `vkCmdBindDescriptorSets`를 호출하는 것은, "자, 아까 말했던 0번 서가에 꽂을 실제 책은 바로 이것이다"라고 실제 `VkDescriptorSet`을 파이프라인에 연결해주는 행위입니다. 즉, `Layout`은 약속/설계도이고, `Set`은 그 약속을 이행하는 실제 객체입니다.

### 5. 시험 대비 핵심 요약

- **데이터의 종류와 버퍼**:
    - **정점별 데이터** -> **버텍스 버퍼** (보통 Staging Buffer를 거쳐 `DEVICE_LOCAL`에 저장)
    - **전역 데이터** -> **유니폼 버퍼** (CPU가 자주 업데이트해야 하므로 `HOST_VISIBLE` | `HOST_COHERENT` 메모리에 저장)
- **Staging Buffer의 존재 이유**: CPU가 직접 쓰기 힘든 **고성능 GPU 전용 메모리(`DEVICE_LOCAL`)**로 데이터를 효율적으로 전송하기 위한 **중간 다리** 역할.
- **디스크립터는 왜 복잡한가?**: 셰이더와 실제 GPU 리소스를 분리(Decoupling)하여 유연성과 재사용성을 높이기 위함입니다. "하드코딩된 주소" 대신 "주소록"을 사용하는 것과 같습니다.
- **디스크립터 5단계 요약**:
    1.  **Layout (설계도)**: 어떤 종류의 리소스를 쓸지 약속.
    2.  **Pool (메모리 풀)**: 디스크립터를 할당할 공간 확보.
    3.  **Allocate Set (할당)**: 실제 디스크립터 셋 객체 생성.
    4.  **Update Set (연결)**: 디스크립터가 실제 버퍼/이미지를 가리키도록 정보 업데이트.
    5.  **Bind Set (사용)**: 렌더링 시 이 디스크립터 셋을 사용하라고 GPU에 명령.
- **애니메이션**: 매 프레임 **유니폼 버퍼**의 내용을 CPU에서 새로 계산하고 `memcpy`로 덮어쓰는 과정의 반복입니다.

이러한 비유와 상세 설명을 통해, 각 Vulkan 객체가 단순한 API 호출이 아니라 특정 목적을 가진 유기적인 시스템의 일부임을 이해하는 데 도움이 되기를 바랍니다.



1. 강의 목표 (Objectives)
   * Vulkan의 Vertex Buffer와 Uniform Buffer의 기능 학습
       * Vertex Buffer (정점 버퍼): 정점 정보(위치, 색상 등)를 저장합니다.
       * Uniform Buffer (유니폼 버퍼): 시간, 애니메이션 진행 상태, 변환 행렬(Model-View-Projection) 등
         프로그램 전반에 걸쳐 변하지 않거나 모든 정점에 공통적으로 적용되는 데이터를 저장합니다.
   * 회전하는 사각형 만들기
       * 실습 과제 ex02를 통해 구현해 봅니다.

  2. Vertex Buffers (정점 버퍼)

  기본 개념 및 구조
   * Vertex Input: 정점 데이터를 셰이더로 전달하기 위한 입력입니다.
   * GLM: 벡터(vec2, vec3)나 행렬(mat4) 같은 선형 대수 타입을 제공하는 라이브러리를 사용하여 정점
     구조체(struct Vertex)를 정의합니다.
   * Binding & Attributes: Vulkan에게 정점 데이터를 셰이더에 어떻게 전달할지 알려주기 위해 두 가지
     구조체가 필요합니다.
       1. `VkVertexInputBindingDescription`: 데이터를 가져올 간격(stride)과 데이터가
          정점별(Vertex)인지 인스턴스별(Instance)인지 결정합니다.
       2. `VkVertexInputAttributeDescription`: 데이터의 속성(위치, 색상 등)이 버퍼 내 어디에
          위치하는지(offset), 데이터 타입(format)은 무엇인지 설명합니다.

  버퍼 생성 및 메모리 관리 (중요)
  Vulkan에서 버퍼를 만드는 과정은 다음과 같이 세분화되어 있습니다.
   1. 버퍼 객체 생성 (`vkCreateBuffer`): 버퍼의 크기(size), 용도(usage), 공유 모드(sharingMode)를
      설정하여 빈 껍데기(핸들)를 만듭니다. Vulkan은 버퍼를 만들 때 자동으로 메모리를 할당해주지
      않습니다.
   2. 메모리 요구사항 확인 (`vkGetBufferMemoryRequirements`): 생성된 버퍼가 필요로 하는 메모리 크기와
      정렬(alignment), 호환되는 메모리 타입 비트 필드를 확인합니다.
   3. 메모리 타입 찾기: GPU가 제공하는 메모리 힙 중에서 버퍼의 요구사항을 만족하고, 우리가 원하는
      속성(예: CPU에서 접근 가능 HOST_VISIBLE)을 가진 메모리 타입을 찾습니다.
   4. 메모리 할당 (`vkAllocateMemory`): 실제 GPU 메모리(또는 공유 메모리)를 할당합니다.
   5. 버퍼와 메모리 바인딩 (`vkBindBufferMemory`): 할당받은 메모리를 버퍼 객체와 연결합니다.

  데이터 전송 (Staging Buffer)
   * `vkMapMemory` / `memcpy` / `vkUnmapMemory`: CPU 메모리 포인터를 얻어와서 데이터를 복사하고 연결을
     해제합니다.
   * 효율적인 메모리 사용 (Staging Buffer):
       * GPU 전용 메모리(DEVICE_LOCAL)는 CPU에서 직접 접근할 수 없지만 속도가 가장 빠릅니다.
       * 따라서 CPU가 접근 가능한 임시 버퍼(Staging Buffer)에 데이터를 먼저 복사한 후, 이를 다시
         고속의 GPU 전용 버퍼(Local Buffer)로 복사(vkCmdCopyBuffer)하는 방식을 사용합니다.

  명령어 (Commands)
   * `vkCmdBindVertexBuffers`: 렌더링 명령을 기록할 때 사용할 정점 버퍼를 바인딩합니다.
   * `vkCmdDraw`: 정점을 그려라는 명령을 내립니다.

  3. Uniform Buffers (유니폼 버퍼)

  개념
   * Uniform 변수: 셰이더 내에서 읽기 전용으로 사용되는 전역 변수 같은 개념입니다. Vulkan에서는 이를
     Uniform Buffer Object (UBO) 형태로 관리합니다.
   * Descriptor (디스크립터): 셰이더가 버퍼나 텍스처 같은 리소스에 접근할 수 있도록 해주는 연결
     고리입니다.

  구성 단계
   1. Descriptor Set Layout (`VkDescriptorSetLayout`): 셰이더에서 몇 번 바인딩(binding=0)에 어떤
      타입(UNIFORM_BUFFER)의 리소스가 들어올지 '형식'을 정의합니다.
   2. Pipeline Layout: 파이프라인 생성 시 이 레이아웃 정보를 알려줍니다.
   3. Descriptor Pool (`VkDescriptorPool`): Descriptor Set을 할당하기 위한 메모리 풀을 생성합니다.
   4. Descriptor Sets (`VkDescriptorSet`): 실제 리소스 정보를 담을 그릇(Set)을 풀에서 할당받습니다.
   5. Descriptor Update (`vkUpdateDescriptorSets`): 생성된 Set에 실제 버퍼
      정보(VkDescriptorBufferInfo)를 연결(Write)합니다.

  실행 및 동기화
   * 프레임별 업데이트: 화면이 회전하거나 애니메이션이 있다면 매 프레임마다 UBO의 데이터(행렬 등)를
     갱신해야 합니다 (updateUniformBuffer).
   * `vkCmdBindDescriptorSets`: 그리기 명령 전에 해당 Descriptor Set을 바인딩하여 셰이더가 UBO
     데이터를 볼 수 있게 합니다.

  요약
  이 자료는 Vulkan에서 화면에 도형을 그리기 위해 데이터를 어떻게 GPU로 보내고 관리하는지에 대한 핵심
  내용을 담고 있습니다.
   * Vertex Buffer는 도형의 모양(점의 위치) 데이터를 전달하는 통로입니다. 성능을 위해 Staging Buffer를
     활용한 메모리 전송 기법을 사용합니다.
   * Uniform Buffer는 도형의 위치 변환(회전, 이동)이나 카메라 시점 같은 공통 데이터를 전달하는
     통로이며, Descriptor라는 시스템을 통해 셰이더와 연결됩니다.
